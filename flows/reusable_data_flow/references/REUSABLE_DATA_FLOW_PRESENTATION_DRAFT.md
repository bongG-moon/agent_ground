# AgentHub 우수사례 발표 초안

## 발표 제목 제안

**업무 데이터 연결에서 시작한, 누구나의 Agent 개발**  
유연 데이터 조회 Flow와 AgentHub 확장 방향

- 발표 시간: 10~15분
- 권장 장표 수: 8장
- 발표 대상: Langflow와 Agent 개발을 잘 모르는 사람도 포함
- 핵심 메시지: **업무를 가장 잘 아는 사람이 공유된 Flow를 조합해 직접 Agent를 만들 수 있도록, 가장 먼저 데이터 조회의 진입장벽을 낮췄다.**

---

## 전체 스토리라인

1. 왜 Langflow를 선택했는가: 누구나 개발할 수 있는 환경
2. 왜 첫 Flow로 데이터 조회를 선택했는가: 입문자의 첫 성공 경험
3. Flow는 어떻게 구성되는가: 등록과 조회의 두 단계
4. 1단계 상세: 데이터 소스 사용설명서 만들기
5. 2단계 상세: 자연어를 실제 조회와 공통 결과로 연결하기
6. 가장 어려웠던 부분: 유연한 요청과 안정적인 실행의 균형
7. Agent Builder를 사용해 수월했던 부분
8. AgentHub를 통해 어떻게 확산할 것인가

# 1. 왜 Langflow였나: 누구나 Agent를 만들 수 있는 환경

## 슬라이드 핵심 문장

**Agent 개발을 일부 개발자의 영역이 아니라, 업무를 아는 사람이 직접 참여할 수 있는 영역으로 만들고 싶었다.**

## 슬라이드에 넣을 문구

**내가 기대한 환경**

- 코드를 몰라도 적당한 교육과 관심이 있으면 시작할 수 있다.
- 화면에서 Component를 연결하며 동작을 이해하고 바로 시험할 수 있다.
- 완성한 Flow를 JSON으로 공유하고, 다른 사람이 가져와 활용할 수 있다.

**Langflow를 선택한 이유**

- 시각적 편집기로 Prompt, LLM, 데이터 소스, Agent 등을 연결할 수 있다.
- Playground에서 만들면서 바로 실행하고 결과를 확인할 수 있다.
- Flow를 내보내고 다시 불러올 수 있어 AgentHub의 공유 방식과 잘 맞는다.

> Langflow가 공개한 방향도 AI Agent 개발을 빠르고 유연하며 누구에게나 접근 가능하게 만들고, 시각적 방식으로 진입장벽을 낮추는 데 있다.

## 이미지·도형 구성 제안

- 화면 중앙에 `업무 사용자` 아이콘을 두고, 오른쪽으로 `Langflow`, `AgentHub`, `나만의 Agent`가 이어지는 큰 화살표를 배치한다.
- `Langflow` 아래에는 작은 아이콘 3개와 함께 `시각적 구성`, `즉시 테스트`, `JSON 공유`만 표시한다.
- 배경 이미지 대신 실제 Langflow 빈 Canvas 또는 간단한 Flow 화면을 오른쪽 40% 영역에 넣는다. 노드 글자가 읽히지 않을 정도로 축소하지 말고 4~6개 노드만 보이는 화면이 적당하다.
- 하단에는 한 줄로 **"누구나 개발할 수 있는 환경"**을 강조한다.
- Langflow 공식 화면을 사용할 경우 작은 출처 표기를 넣고, 가능하면 직접 실행 중인 Langflow 화면을 캡처해 사용한다.

## 발표자 노트

제가 Langflow를 선택한 이유는 코딩을 조금 할 줄 아는 사람만 Agent를 만드는 환경을 원하지 않았기 때문입니다. 업무를 잘 아는 사람이 적당한 교육을 받고 Component를 연결해 직접 필요한 Agent를 만드는 모습을 기대했습니다. Langflow의 공식 설명에서도 시각적 편집기를 통한 빠른 구성과 테스트, 공유를 강조하고 있습니다. 특히 Flow를 JSON으로 내보내고 가져올 수 있다는 점은 각자가 만든 결과를 AgentHub에서 공유하고 재사용하는 방식과 잘 맞았습니다.

# 2. 왜 첫 Flow가 데이터 조회였나

## 슬라이드 핵심 문장

**입문자가 Agent에 관심을 갖는 가장 빠른 방법은, 자신이 실제로 쓰는 데이터가 움직이는 경험이라고 생각했다.**

## 슬라이드에 넣을 문구

**출발점이 된 질문**

> Agent Builder 입문자가 어렵지 않게 시작하면서도, 바로 업무 가치를 느끼게 하려면 무엇이 필요할까?

**이미 쉬워진 것과 아직 어려운 것**

| 비교적 쉽게 연결 가능 | 여전히 코드를 알아야 하는 영역 |
| --- | --- |
| LLM, 큐브, 메일링 등 | SQL, API 요청값, 문서 ID, 인증 방식 등 실제 업무 데이터 조회 |

**정확히 풀고 싶었던 문제**

> 코딩에 익숙하지 않은 사용자가 조회 코드를 수정하지 않고, 자연어만으로 필요한 데이터 소스와 조건을 선택해 실행하고, 그 결과를 다른 Flow에서도 재사용할 수 있게 한다.

**그래서 한 일**

- 자주 사용하던 데이터 조회 코드를 소스별 Component로 만들었다.
- 사용자는 코드 대신 원하는 업무를 자연어로 설명하도록 했다.
- LLM은 적합한 데이터와 조건을 찾고, Flow가 실제 조회를 수행하도록 역할을 나눴다.

## 이미지·도형 구성 제안

- 슬라이드를 좌우 `Before / After` 구조로 만든다.
- 왼쪽에는 SQL, API, 인증키를 나타내는 3개 작은 코드 문서 아이콘과 `데이터마다 다른 조회 방식`을 배치한다.
- 오른쪽에는 말풍선 하나에 `5월 20일 FAB1 D/A1의 DDR5 생산실적 보여줘`를 넣고, 화살표 끝에 표 형태의 조회 결과를 배치한다.
- 중앙 화살표 위에는 **"코드 수정 없이"**를 강조한다.
- 별도의 인물 사진이나 추상적인 AI 이미지는 사용하지 않는다. 이 장은 문제와 변화가 한눈에 보여야 한다.

## 발표자 노트

LLM이나 메일링 같은 기능은 Langflow에서 비교적 쉽게 연결할 수 있었습니다. 하지만 실제로 내가 매일 보던 데이터를 가져오려면 SQL, API Body, 문서 ID처럼 데이터마다 다른 코드를 알아야 했습니다. 입문자가 자신의 업무 데이터로 첫 결과를 얻을 수 있다면, 그다음에는 분석이나 보고서 같은 Flow도 자연스럽게 만들 수 있다고 봤습니다. 그래서 첫 번째 공통 기능으로 데이터 조회를 선택했습니다.

# 3. 한눈에 보는 Flow: 등록과 조회의 두 단계

## 슬라이드 핵심 문장

**복잡해 보이는 전체 Flow도 사용 관점에서는 "데이터를 등록하는 단계"와 "말로 조회하는 단계" 두 가지다.**

## 슬라이드에 넣을 문구

### 1단계. 데이터 소스 등록·관리

`데이터 설명 입력 → source_catalog 생성 → 저장·재사용`

- 어떤 데이터인지
- 어떤 조건이 필요한지
- 실제로 어떻게 조회하는지

### 2단계. 자연어 기반 데이터 조회

`자연어 요청 → 실행 정보 생성 → 소스 조회 → 공통 결과 출력`

- 사용자는 자연어만 입력
- Flow가 맞는 소스와 Component 선택
- 결과는 다른 Flow가 읽을 수 있는 형태로 출력

## 이미지·도형 구성 제안

- 이 장의 70% 이상을 **실제 Langflow 전체 Flow 캡처 이미지**로 사용한다.
- 캡처 이미지 위에 반투명 테두리 박스 두 개를 겹쳐 표시한다.
  - 1단계 영역: 청록색 테두리, 라벨 `데이터 소스 등록·관리`
  - 2단계 영역: 파란색 테두리, 라벨 `자연어 기반 데이터 조회`
- 화면이 복잡하면 중요하지 않은 노드는 40~50% 흐리게 처리하고, 두 단계의 경계와 진행 방향만 선명하게 둔다.
- 하단에는 `등록은 한 번, 조회는 여러 번`이라는 짧은 문장을 넣는다.
- 실제 화면을 캡처할 때 노드 이름이 보이도록 `Zoom to Fit` 후 불필요한 Canvas 여백을 잘라낸다.

## 발표자 노트

전체 Flow에는 Component가 많지만 사용 관점에서는 두 단계입니다. 먼저 데이터 소스를 한 번 등록하고, 이후 사용자는 자연어로 여러 번 조회합니다. 다음 두 장에서 각 단계를 확대해 보겠습니다. 이 구조를 분리한 이유는 조회할 때마다 SQL이나 API 설정을 다시 만들지 않고, 등록된 정보를 반복해서 안전하게 사용하기 위해서입니다.

# 4. 1단계: 데이터 소스의 사용설명서를 만든다

## 슬라이드 핵심 문장

**source_catalog는 LLM과 Flow가 함께 읽는 "데이터 소스 사용설명서"다.**

## 슬라이드에 넣을 문구

```text
사람의 설명
→ Prompt + LLM Caller
→ 07 Catalog Normalizer
→ source_catalog
→ 09 Store / 10 Loader
```

**사용자가 설명하는 내용**

- 데이터의 이름과 용도
- 검색에 필요한 조건과 형식
- 어떤 질문에서 이 데이터를 사용하면 되는지
- SQL, API URL, 문서 ID 등 조회 설정

**주요 Component**

| Component | 하는 일 |
| --- | --- |
| `08 LLM Caller` | 사람이 적은 설명을 LLM에 전달 |
| `07 Catalog Normalizer` | 설명을 표준 `source_catalog`로 정리 |
| `09 MongoDB Store` | 긴 조회 설정을 포함한 Catalog 저장 |
| `10 MongoDB Loader` | 저장한 Catalog를 다시 불러와 재사용 |

**중요한 분리**

계정, Token 같은 인증 정보는 Catalog가 아니라 실제 데이터 Component의 입력에만 둔다.

## 이미지·도형 구성 제안

- 상단에는 위 흐름을 5개의 가로형 도형으로 표현하고, 실제 Flow의 1단계 영역 캡처를 배경 또는 하단에 작게 붙인다.
- `source_catalog` 도형은 일반 박스보다 작은 문서 모양으로 만들고 안에 다음 4줄만 보여준다.
  - `name: production_summary`
  - `keywords: 생산, 실적`
  - `required: 날짜, 공장, 공정`
  - `source: Oracle`
- `인증 정보는 별도 관리`는 자물쇠 아이콘이 있는 작은 주석으로 표시한다.
- JSON 전체를 화면에 넣지 않는다. 발표자가 설명할 수 있는 필드만 남긴다.

## 발표자 노트

첫 단계는 데이터 소스의 사용설명서를 만드는 과정입니다. 사용자는 데이터가 무엇이고 어떤 조건으로 조회하는지를 설명하고, Catalog Normalizer가 이를 정해진 구조로 바꿉니다. 긴 SQL처럼 화면에서 잘릴 수 있는 정보는 MongoDB에 저장해 다시 사용합니다. 인증 정보는 공유 대상인 Catalog와 분리해 각 데이터 Component에서 관리합니다. 이렇게 만들어진 source_catalog는 다음 단계에서 LLM이 소스를 고를 때와 실제 조회 정보를 채울 때 모두 사용됩니다.

# 5. 2단계: 말로 요청하면 실제 조회와 공통 결과까지

## 슬라이드 핵심 문장

**사용자의 한 문장을 "조회 지시서"로 바꾸고, 어떤 데이터 소스든 같은 결과 형태로 돌려준다.**

## 먼저 설명할 용어

> **실행 정보(Payload)**: Component 사이에 전달되는 데이터 묶음. 여기에는 조회 대상, 조건, 실행 설정, 조회 결과가 들어간다.

발표 중에는 `Payload`보다 **"실행 정보"** 또는 **"조회 지시서"**라는 표현을 우선 사용한다.

## 슬라이드에 넣을 문구

### 한눈에 보는 실행 흐름

```text
① 자연어 요청
→ ② 조회 지시서 생성
→ ③ 맞는 데이터 Component만 실행
→ ④ 공통 결과(data_json) 출력
```

**① 사용자의 말**

`5월 20일 FAB1 D/A1 공정의 DDR5 생산실적을 보여줘`

**② LLM + `00 Data Request Normalizer`**

```text
데이터: production_summary
조건: DATE, FACTORY, OPER_NAME, MODE
조회 방식: Oracle
필요 설정: Catalog에서 자동 보강
```

**③ `01~04 Data Nodes`**

- Oracle, H-API, Datalake, Goodocs가 같은 요청을 받는다.
- 이 중 요청에 맞는 Component만 실행하고 나머지는 건너뛴다.

**④ `05 Merger` + `06 Output Builder`**

```json
{
  "success": true,
  "data_result": [[{"WORK_DT": "20260520", "PRODUCTION": 1000}]],
  "source_results": [{"name": "production_summary", "row_count": 1}]
}
```

- `data_result`: 실제 조회된 데이터
- `source_results`: 어떤 소스와 조건으로 몇 건을 조회했는지에 대한 설명
- 동일한 결과를 Chat Output, HTML Report, 다른 Agent의 입력으로 연결 가능

## 이미지·도형 구성 제안

- 화면을 가로 4단계로 나누고 각 단계에 큰 번호를 붙인다.
- 1단계는 말풍선, 2단계는 체크리스트 형태의 `조회 지시서`, 3단계는 Oracle/H-API/Datalake/Goodocs 네 개의 작은 소스 박스, 4단계는 표와 JSON 아이콘으로 표현한다.
- 3단계에서 실제 실행되는 Oracle 박스만 진한 색으로 두고 나머지는 회색으로 표시하면 `맞는 Component만 실행`이 바로 이해된다.
- 4단계 결과에서 JSON 전체를 강조하지 말고 `실제 데이터`와 `조회 설명` 두 영역을 색으로 구분한다.
- 하단에 `한 번 정한 출력 형식 → 여러 Flow에서 재사용` 화살표를 두고 `Chat`, `HTML Report`, `다른 Agent` 아이콘으로 분기한다.

## 발표자 노트

여기서 Payload는 특별한 기술이 아니라 Component 사이에 넘기는 실행 정보 묶음입니다. 사용자의 문장에서 LLM은 데이터 이름과 조건만 고릅니다. `00 Data Request Normalizer`는 source_catalog를 기준으로 실제 조회 방식과 필요한 설정을 채워 안정적인 조회 지시서를 만듭니다. 각 데이터 Component는 자신의 요청이 아니면 실행하지 않습니다. 마지막에는 실제 데이터와 조회 정보를 공통 JSON으로 정리하기 때문에, 결과를 화면에 보여주는 데서 끝나지 않고 HTML 보고서나 다른 Agent로 바로 넘길 수 있습니다.

# 6. 가장 어려웠던 부분: 자유로운 요청을 안정적으로 실행하기

## 슬라이드 핵심 문장

**가장 어려웠던 부분은 "LLM의 유연함"과 "실행 정보의 안정성" 사이의 경계를 정하는 일이었다.**

## 슬라이드에 넣을 문구

| 문제 | 해결 방식 |
| --- | --- |
| 같은 의미도 사람마다 다르게 표현 | LLM은 `데이터 이름 + 조건값`만 고르고, Normalizer가 표준 형식으로 정리 |
| LLM이 SQL, API 주소, 문서 ID를 추측할 가능성 | 실행 설정은 LLM이 만들지 않고 `source_catalog`에서만 가져옴 |
| 입력이 Text, Message, JSON 등 여러 형태로 전달 | `00 Data Request Normalizer`에서 다양한 입력을 하나의 구조로 변환 |
| 긴 SQL이 화면 미리보기에서 잘릴 수 있음 | 원본 데이터 연결 또는 MongoDB Store/Loader로 전체 내용을 보존 |
| 여러 소스를 조회하면 결과 순서와 출처를 잃기 쉬움 | `05 Merger`가 요청 순서와 `source_results`를 함께 보존 |
| 공유되는 Flow에 인증 정보가 포함될 위험 | 계정과 Token은 Catalog·LLM 출력과 분리해 데이터 Component에만 입력 |

## 이미지·도형 구성 제안

- 장표를 정확히 50:50으로 나누고 왼쪽은 `문제`, 오른쪽은 `해결 방식`으로 구성한다.
- 각 행의 문제와 해결을 같은 높이에 맞추고 가는 화살표로 연결한다.
- 왼쪽은 옅은 회색 또는 주황 계열, 오른쪽은 청록 또는 파랑 계열을 사용한다.
- 중앙에 별도의 복잡한 Flow 그림을 넣지 않는다. 이 장은 비교표 자체가 핵심 시각 자료다.
- 제목 아래에 작은 글씨로 `실행 정보(Payload) = 조회를 위해 Component 사이에 전달되는 데이터 묶음`을 다시 한번 표시한다.

## 발표자 노트

자연어는 자유로워야 하지만 실제 데이터 조회는 정확해야 합니다. 그래서 LLM에게 모든 것을 맡기지 않고 역할의 경계를 정했습니다. LLM은 사용자의 의도를 이해해 데이터와 조건을 고르고, 실행에 필요한 SQL이나 API 정보는 사람이 관리하는 Catalog에서 가져옵니다. Normalizer가 이 둘을 합쳐 표준 형태로 만들면서 유연함과 안정성을 동시에 확보했습니다.

# 7. Agent Builder를 활용해 수월했던 부분

## 슬라이드 핵심 문장

**Langflow는 코드를 없애는 도구라기보다, 기존 기능을 보이는 블록으로 바꾸고 빠르게 조합하게 해주는 도구였다.**

## 슬라이드에 넣을 문구

### 1. 기존 조회 코드의 재사용

- 이미 자주 쓰던 Oracle, H-API, Datalake, Goodocs 조회 코드를 Component로 감쌀 수 있었다.
- 처음부터 데이터 시스템을 새로 만들 필요 없이 역할별 블록으로 나눌 수 있었다.

### 2. 연결과 역할을 눈으로 확인

- 어느 Component가 어떤 값을 받고 어디로 넘기는지 화면에서 확인할 수 있었다.
- 소스별 Component가 자기 요청만 실행하게 하니 연결 구조도 단순해졌다.

### 3. 빠른 시험과 확장

- Dummy 데이터와 Playground로 외부 시스템을 호출하지 않고도 흐름을 먼저 확인할 수 있었다.
- 최종 출력을 `data_json`으로 통일하니 HTML Report 같은 다른 Flow와 연결하기 쉬웠다.

## 이미지·도형 구성 제안

- 세로 카드 3개보다 **하나의 가로 Flow 이미지에 3개의 Callout**을 붙이는 구성이 더 좋다.
- Callout은 각각 `기존 코드 → Component`, `시각적 연결`, `다른 Flow로 확장`을 가리킨다.
- 실제 Langflow 화면에서 Oracle Data, Merger, Output Builder가 보이는 부분을 잘라 사용한다.
- `Component화`, `눈으로 검증`, `재사용` 세 단어만 굵게 표시하고 설명은 2줄 이내로 유지한다.

## 발표자 노트

생각보다 수월했던 이유는 이미 업무에서 반복해서 쓰던 조회 코드가 있었기 때문입니다. 이를 Component로 감싸고 역할별로 연결하니 전체 구조가 눈에 보였습니다. Dummy 데이터로 먼저 연결을 시험하고, 출력 형식을 통일하니 다른 Flow로 확장하는 작업도 쉬워졌습니다. 이 경험은 AgentHub에 공유할 때도 중요합니다. 사용자는 내부 코드를 모두 이해하지 않아도 Flow의 역할과 연결 지점을 보고 활용할 수 있기 때문입니다.

# 8. AgentHub와 함께 만드는 다음 단계

## 슬라이드 핵심 문장

**유연 데이터 조회 Flow는 하나의 완성품이 아니라, 사람들이 자신의 업무 Agent를 만들게 하는 첫 번째 공통 블록이다.**

## 슬라이드에 넣을 문구

### 앞으로의 활성화 방향

```text
실제 업무 분석
→ 반복되는 공통 기능 정의
→ 재사용 가능한 Component·Flow 구현
→ AgentHub에 JSON·사용 예시와 함께 공유
→ 현업 사용과 피드백
→ 다음 공통 기능 발굴
```

### AgentHub에 기대하는 역할

- **발견**: 내 업무와 비슷한 검증된 Flow를 쉽게 찾는 곳
- **재사용**: JSON을 가져와 데이터와 조건만 바꿔 시작하는 곳
- **학습**: 다른 사람의 구성 방식과 사용 예시를 보며 Agent를 배우는 곳
- **확산**: 개인의 아이디어가 여러 사람의 공통 업무 도구로 발전하는 곳

### 앞으로 공유할 Flow의 예

- 업무를 설명하면 Agent 구조를 설계해주는 Flow
- 데이터와 보고서 양식을 주면 HTML Report를 만드는 Flow
- 내용을 입력하면 카드 뉴스를 만드는 Flow
- 유연 데이터 조회 Flow와 위 Flow들을 연결한 `조회 → 분석 → 보고서·콘텐츠 생성` 조합

**기대하는 변화**

> 매번 처음부터 개발하는 방식에서, AgentHub의 공통 기능을 찾아 조합하고 자신의 업무에 맞게 완성하는 방식으로

## 이미지·도형 구성 제안

- 중앙에 5단계 순환형 도형을 배치한다: `업무 분석 → 공통 기능 → Flow 제작 → AgentHub 공유 → 현업 활용·피드백`.
- `AgentHub 공유` 도형을 가장 크게 표시하고, 그 주변에 `발견·재사용·학습·확산` 네 개의 작은 라벨을 둔다.
- 하단에는 작은 블록 4개를 연결한다: `데이터 조회`, `Agent 설계`, `HTML Report`, `카드 뉴스`.
- 첫 번째 `데이터 조회` 블록에는 `현재 우수사례` 표시를 붙이고, 나머지 블록으로 화살표가 이어지게 한다.
- 마지막 문장은 하단 전체 폭을 사용해 크게 배치한다: **"공유되는 Flow가 늘수록, Agent 개발은 처음부터 만드는 일보다 필요한 기능을 조합하는 일에 가까워진다."**

## 발표자 노트

앞으로는 사람들이 실제로 반복하는 업무를 분석하고, Agent로 만들 때 공통으로 사용되는 기능을 먼저 정의해 구현할 생각입니다. AgentHub는 단순히 JSON 파일을 보관하는 곳이 아니라, 검증된 Flow를 발견하고 가져와 학습하며 다시 개선된 결과를 공유하는 공간이 되어야 합니다. 유연 데이터 조회 Flow는 그 첫 번째 공통 블록입니다. 이런 블록이 쌓이면 코드를 모르는 사람도 자신에게 필요한 기능을 선택하고 연결해 Agent를 만들 수 있습니다.

## 장표 제작 공통 가이드

- 화면 비율은 16:9를 기준으로 한다.
- 한 장에 핵심 메시지는 한 문장만 두고, 본문은 발표자가 읽는 대본이 아니라 키워드 중심으로 줄인다.
- 3·4·5·7장은 실제 Langflow 화면을 사용해 이 Flow가 구현된 결과임을 보여준다.
- 전체 Flow 화면, 1단계 영역, 2단계 영역을 각각 캡처해 같은 이미지 톤으로 사용한다.
- 1단계와 2단계 색은 모든 장에서 일관되게 유지한다.
  - 1단계 데이터 등록: 청록색
  - 2단계 데이터 조회: 파란색
  - AgentHub와 확산: 기존 AgentHub 브랜드 색상 우선
- 기술 용어는 처음 나올 때만 쉬운 말과 함께 표시한다.
  - `source_catalog`: 데이터 소스 사용설명서
  - `Payload`: Component 사이에 전달되는 실행 정보
  - `data_json`: 다른 Flow도 읽을 수 있는 공통 결과
- 코드나 JSON은 설명 대상인 필드만 보여주고, 한 화면에 8줄을 넘기지 않는다.
- 발표 자료 우측 하단에는 `AgentHub 우수사례 | 유연 데이터 조회 Flow`를 공통 꼬리말로 넣는다.

## 준비할 이미지 체크리스트

1. Langflow 전체 Flow 화면
2. 데이터 소스 등록·관리 영역 확대 화면
3. 자연어 조회·데이터 Component·결과 영역 확대 화면
4. Playground에서 자연어 질문과 조회 결과가 함께 보이는 화면
5. AgentHub에서 Flow/JSON이 등록되어 있거나 공유되는 화면이 있다면 마지막 장에 사용할 캡처

AgentHub 화면이 아직 발표용으로 정리되지 않았다면, 마지막 장은 실제 화면 대신 순환형 도형을 사용한다. 플랫폼 화면과 발표 메시지가 다르면 오히려 시선이 분산될 수 있다.
