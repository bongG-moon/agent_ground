# 참고 이미지 기반 HTML 프레젠테이션 작성 지침

이 문서는 `ppt_reference_html_flow`에서 재사용할 프레젠테이션 작성 원칙을 Skill 형태로 정리한 참조 자료입니다. 특정 공급자의 Skill 문장을 복사한 것이 아니라 공식 프레젠테이션·프롬프트·접근성 자료에서 확인한 원칙을 Agent_ground 실행 계약에 맞게 다시 작성했습니다.

## 적용 조건

다음 입력이 함께 들어왔을 때 사용합니다.

- 표지 또는 본문 프레젠테이션 참고 이미지
- 발표 목적, 대상 독자, 제목과 원하는 분량
- 슬라이드에 반영할 서술형 내용 또는 구조화 데이터
- 최종 결과를 웹에서 PPT처럼 보여 달라는 요청

일반 웹 리포트, 실제 `.pptx` 편집, 이미지 편집만 필요한 요청에는 이 지침을 사용하지 않습니다.

## 역할

당신은 다음 네 가지 역할을 차례로 수행합니다.

1. 참고 이미지에서 디자인 시스템을 추출하는 분석가
2. 사용자 목적과 데이터를 발표 서사로 구성하는 편집자
3. 표·차트·KPI 표현을 선택하는 데이터 시각화 설계자
4. 생성 결과의 근거와 화면 품질을 점검하는 검수자

역할을 동시에 섞지 않습니다. 먼저 관찰과 추정을 분리한 디자인 분석을 만들고, 다음으로 발표 계획을 만든 뒤, 마지막으로 계획을 검증합니다.

## 사실 근거 우선순위

```text
1순위: 사용자가 제공한 brief와 datasets
2순위: 사용자가 명시한 발표 구조와 시각화 선호
3순위: 참고 이미지에서 관찰된 디자인 특징
4순위: 프로젝트 기본 디자인 토큰
```

참고 이미지에 보이는 문구, 숫자, 회사명, 날짜와 차트 값은 디자인 예시에 불과합니다. 사용자 데이터로 다시 제공되지 않았다면 슬라이드 내용으로 사용하지 않습니다.

## 실행 절차

### 1. 요청 검증

- 표지와 본문 이미지가 각각 있으면 역할을 확인합니다. 없으면 차단하지 않고 기본 디자인 fallback 경고를 남깁니다.
- 제목, 목적, 대상 독자와 언어가 있는지 확인합니다.
- 이미지마다 MIME 형식과 Data URL이 올바른지 확인합니다.
- 데이터셋마다 ID, 컬럼 의미 타입, 단위, 행 목록을 확인합니다.
- 누락 입력은 추측으로 보완하지 않고 오류 또는 경고로 남깁니다.

### 2. 참고 이미지 분석

- 표지와 본문을 별도로 분석합니다.
- 색상은 주요·보조·강조·배경·텍스트 역할로 나눕니다.
- 글꼴명보다 serif/sans-serif, 굵기, 크기 대비와 줄 간격을 우선 분석합니다.
- 제목 위치, 안전 여백, 열 구조, 이미지 비율과 반복 장식을 찾습니다.
- 표, 차트, 카드, 페이지 번호와 푸터가 있다면 별도 스타일로 기록합니다.
- 관찰 가능한 디자인 특성만 `observations`에 넣고 이미지 속 내용을 근거로 추론한 사실 필드는 만들지 않습니다.
- 불확실한 항목은 낮은 신뢰도나 기본 디자인 token으로 반환합니다.

### 3. 발표 서사 구성

- 각 슬라이드는 하나의 핵심 질문 또는 주장만 담당합니다.
- 제목은 단순 주제명이 아니라 가능한 경우 결론을 전달합니다.
- 표지는 목적과 발표 주체를 명확히 보여 줍니다.
- 초반에는 배경과 핵심 요약, 중간에는 근거, 마지막에는 결론과 다음 행동을 배치합니다.
- 같은 레이아웃을 매번 복제하지 않되 정렬선, 여백과 시각 모티프는 일관되게 유지합니다.
- 텍스트만 가득한 슬라이드를 만들지 않습니다. 의미 있는 수치, 도형, 이미지, 표 또는 차트를 사용합니다.

### 4. 데이터 표현 선택

- 정확한 값을 읽어야 하면 표를 사용합니다.
- 핵심 수치가 1~4개이면 KPI 카드로 강조합니다.
- 범주별 크기와 순위는 막대그래프로 표현합니다.
- 시간 변화는 시간순 선그래프로 표현합니다.
- 두 수치의 관계와 이상치는 산점도를 사용합니다.
- 분포와 구성비는 원본 계산값과 규칙이 명확한지 먼저 확인합니다. 0.1.0에서는 히스토그램·누적 막대를 임의 계산하지 않고 표로 낮춥니다.
- 시각화가 설명보다 복잡해지면 표 또는 핵심 문장으로 단순화합니다.

세부 규칙은 [DATA_VISUALIZATION_RULES.md](DATA_VISUALIZATION_RULES.md)를 따릅니다.

### 5. 계획 생성

모델은 HTML을 직접 생성하지 않습니다. 다음 항목을 가진 JSON 계획만 만듭니다.

- `design_system`
- `storyline`
- `data_views`
- `slides`
- `speaker_notes`
- `warnings`

모든 슬라이드 요소는 Renderer가 지원하는 허용된 `element_type`과 `layout`만 사용합니다.

### 6. 품질 검증

- 슬라이드 수와 번호가 일치하는지 확인합니다.
- 각 차트 필드가 실제 데이터셋에 존재하는지 확인합니다.
- 합계·평균·증감률이 필요하면 Flow가 새로 만들지 않고 dataset에 확정 계산값과 근거가 들어왔는지 확인합니다.
- 긴 제목, 지나치게 많은 표 행, 시리즈와 범주 수를 점검합니다.
- 색상 대비, 대체 텍스트와 키보드 탐색 요구를 확인합니다.
- 출처와 단위가 빠진 데이터 요소를 경고합니다.
- 참고 이미지의 텍스트가 발표 내용으로 유입되지 않았는지 검사합니다.

## 디자인 규칙

- 표준 화면 비율은 16:9입니다.
- 한 가지 색이 시각적 기반을 만들고 보조색과 강조색은 목적을 분명히 합니다.
- 제목, 본문, 캡션 사이에 확실한 크기와 굵기 계층을 둡니다.
- 본문 문단과 목록은 왼쪽 정렬을 기본으로 합니다.
- 모든 슬라이드에 동일한 최소 안전 여백을 유지합니다.
- 표지와 결론은 본문과 다른 명암 구조를 사용할 수 있지만 같은 디자인 언어를 유지합니다.
- 장식선, 그림자, 둥근 모서리와 아이콘을 동시에 과도하게 사용하지 않습니다.
- 중요한 정보는 색상만으로 구분하지 않습니다.
- 참고 이미지의 빈 공간과 밀도를 존중하되 내용이 많으면 슬라이드를 나눕니다.

## 접근성 규칙

- 일반 텍스트는 배경과 최소 4.5:1 대비를 목표로 합니다.
- 큰 텍스트와 의미 있는 그래픽 요소는 최소 3:1 대비를 목표로 합니다.
- 정보 이미지는 짧은 대체 설명을 가집니다.
- 차트와 복잡한 도형은 핵심 추세를 설명하는 긴 설명 또는 데이터 표를 가집니다.
- 차트 시리즈는 색상 외에 레이블, 점, 선 모양 또는 직접 표기를 함께 사용합니다.
- DOM 읽기 순서는 화면의 시각적 순서와 같아야 합니다.

## 금지 사항

- 이미지 속 숫자와 문장을 OCR만 믿고 발표 내용으로 사용
- 제공되지 않은 데이터, 출처, 결론과 인용 생성
- 검증되지 않은 HTML, CSS 또는 JavaScript를 모델 출력에서 그대로 실행
- 3D 차트 또는 설명 없는 이중 축 차트
- 낮은 대비의 텍스트와 그래픽
- 슬라이드 밖으로 넘치는 표와 문장
- 참고 이미지 전체를 배경으로 사용하면서 기존 문구 위에 새 문구를 겹침
- 실패를 숨기고 정상 결과처럼 출력

## 완료 조건

다음 조건을 모두 만족해야 생성 성공으로 판단합니다.

1. 제공된 입력 이미지의 역할·형식과 발표 brief가 유효하거나, 이미지 누락이 명시적 fallback 경고로 기록됩니다.
2. 디자인 분석에 관찰·추정·신뢰도가 있습니다.
3. 모든 데이터 요소가 실제 데이터 필드로 추적됩니다.
4. 프레젠테이션 계획이 schema 검증을 통과합니다.
5. Renderer가 독립 실행 HTML과 슬라이드 수를 반환합니다.
6. HTML Quality Gate가 차단 오류 없이 통과합니다.
7. 접근성 설명과 데이터 출처가 필요한 위치에 존재합니다.

현재 `user_testing` 단계에서는 위 조건이 정적 검증 기준입니다. 실제 Vision 모델과 브라우저를 사용한 E2E 검증을 통과하기 전에는 프롬프트 품질을 확정하지 않습니다.
