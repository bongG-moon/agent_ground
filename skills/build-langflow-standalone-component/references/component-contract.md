# Standalone Component 계약

## Component인지 먼저 판단하기

Standalone은 파일 배포 방식이고 Component는 기능 단위다. 아래 질문에 모두 `예`라고 답할 수 있어야 공용 Component로 등록한다.

1. 특정 Flow를 몰라도 한 문장으로 독립 사용 사례를 설명할 수 있는가?
2. 다른 Flow나 Agent가 직접 연결해 쓸 수 있는 안정된 입력·출력 계약인가?
3. 단순 변환·prompt 조립·demo 값 생성이 아니라 의미 있는 업무 기능을 수행하는가?
4. 오류, 보안과 제한 정책을 독립적으로 정의할 수 있는가?
5. 별도 버전·테스트·설명서를 유지할 가치가 있는가?

하나라도 아니라면 `flows/<flow_id>/nodes/`의 Flow 내부 노드로 관리하고 `internal_nodes.json`에 기록한다. 내부 노드도 한 파일로 실행 가능한 Standalone 포장을 유지하지만 Component registry에는 넣지 않는다.

## 목적 문장

`<입력>`을 받아 `<처리>`하고 `<Langflow 출력 타입>` 하나 또는 명시된 여러 결과로 반환한다.

## 입력 표

| 화면 이름 | name | Input 클래스 | 연결 타입 | 필수 | 기본값 | 설명 |
| --- | --- | --- | --- | --- | --- | --- |
| 사용자에게 보이는 이름 | stable_name | DataInput 등 | Data/Message 등 | 예/아니오 | 안전한 값 | 무엇을 넣고 언제 비워둘 수 있는지 |

입력 설계 시 다음을 확인한다.

- 연결 가능한 데이터인가, 화면에서 입력하는 설정인가?
- secret인가?
- 비어 있을 때 fail-closed인가, 정상 빈 값인가?
- 최대 파일 수, 행 수, byte, timeout 제한이 있는가?
- 고급 설정으로 숨겨도 초보자가 기본값으로 안전하게 쓸 수 있는가?

## 출력 표

| 화면 이름 | name | 타입 | method | 의미 | 다음 연결 |
| --- | --- | --- | --- | --- | --- |
| 사용자에게 보이는 이름 | stable_output | Data | build_result | 성공·오류 계약 | Agent/Parser 등 |

## 설명 문구 기준

- 무엇을 하는지 첫 문장에서 말한다.
- 무엇을 하지 않는지도 부작용 관점에서 말한다.
- 입력 설명에 형식과 실제 예시를 포함한다.
- 출력 설명에 타입과 다음 연결 대상을 포함한다.
- 실패가 빈 성공 결과와 어떻게 다른지 말한다.

## 최소 검증 세트

1. 가장 작은 정상 입력
2. 실제 업무형 정상 입력
3. 필수 입력 누락
4. 잘못된 타입 또는 형식
5. 행·파일·byte 제한 초과
6. 외부 인증·timeout 실패
7. secret 비노출
8. 선언 output method와 실제 method 일치
9. 단독 Python import 또는 격리 로드
10. 관련 Flow 회귀 테스트
